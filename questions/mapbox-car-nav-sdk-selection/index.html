Mapboxでカーナビを作るときのSDK選定

最終更新: 2026-06-07

Mapboxでカーナビアプリを作るときは、1つのSDKですべてを完結させるというより、 地図表示、目的地検索、経路案内、独自ロジックを役割ごとに分けて組み合わせる考え方が実用的です。 結論から言うと、運転中の案内体験を作る中心は Navigation SDK、地図の見た目や独自レイヤー制御の中心は Maps SDK、目的地入力や候補提示は Search SDK または Search / Geocoding API が適しています。

まず結論

  • 地図を表示する: Maps SDK
  • ターンバイターン案内を実装する: Navigation SDK
  • 住所、施設、POI検索を入れる: Search SDK または Search / Geocoding API
  • 独自の配車、配送、車両制約ロジックを組む: バックエンド側で Directions 系 API や独自判定を併用する
  • 独自データを地図に重ねる: Maps SDK + Mapbox Studio / Tiling Service

SDKごとの役割

1. Maps SDK

Maps SDKは、地図を描画して操作するための土台です。 現在地表示、ピン、GeoJSONレイヤー、渋滞表示風の独自表現、配送先一覧、走行軌跡、危険地点オーバーレイなど、 ナビ画面の「見せ方」を細かく作り込みたいならここが中心になります。

ただし、Maps SDKだけではターンバイターン案内や音声案内は完成しません。 地図を描くためのSDKであり、案内体験そのものはNavigation SDKの担当です。

2. Navigation SDK

カーナビを作るなら最重要なのがNavigation SDKです。 ルート取得、リルート、音声案内、案内UI、到着判定、走行中の追従、ルートライン表示など、 一般的なカーナビに必要な要素をまとめて扱えます。

「とりあえず運転ナビとして動かしたい」段階では、まずNavigation SDKを中核に選ぶのが妥当です。 逆に言うと、Maps SDKだけでゼロから案内体験を組むのは実装負荷が高く、保守も重くなります。

3. Search SDK / Search API / Geocoding API

目的地検索は別レイヤーで考えたほうが整理しやすいです。 アプリ内で検索ボックス、候補補完、POI検索を自然に出したいならSearch SDKが向いています。 一方、サーバー側で検索を制御したい、あるいは独自UIからHTTPベースで扱いたいなら Search APIやGeocoding APIが扱いやすいです。

「住所を入れて候補を出す」「店舗名から目的地を選ぶ」「地図の中心点から住所を引く」といった処理は、 Navigation SDK本体よりこちらの担当です。

用途別の選び方

最短で一般的なカーナビを作りたい

Navigation SDK + Search SDK が基本です。 地図描画はNavigation SDK側の機能でかなり賄えるため、最初からMaps SDKを深く触らなくても始められます。 ただし、UIの自由度を強く求めるならMaps SDKの理解は結局必要になります。

地図表現を大きくカスタマイズしたい

Maps SDK + Navigation SDK の組み合わせが向いています。 例として、配送ルート以外に立ち寄り候補、危険物ルート警告、車高制限地点、営業エリア、社内案件データを 重ねたい場合はMaps SDK側のレイヤー制御が重要になります。

検索体験を独自UIで作りたい

フロントにSearch SDKを直接入れる方法と、バックエンドからSearch API / Geocoding APIを叩く方法があります。 モバイルUIに自然に組み込みたいならSDK、検索ログ管理や事業ロジック統合を優先するならAPIベースが扱いやすいです。

トラック、配送、業務車両向けに作り込みたい

ここではMapbox標準SDKだけで完結すると考えないほうが現実的です。 ルート案内の基盤はNavigation SDKで持ちつつ、車両制約、訪問順最適化、社内マスタとの突合、到着業務フローは 自前バックエンドや別ロジック層で補う構成が安定します。

実装判断の目安

やりたいこと 主に使うもの 補足
地図を表示する Maps SDK ベース地図、マーカー、レイヤー制御向け
音声付き経路案内をする Navigation SDK カーナビの中心機能
検索候補を出す Search SDK モバイルUI一体型で使いやすい
住所と座標を変換する Geocoding API バックエンド連携にも向く
独自の地図データを重ねる Maps SDK + Studio / Tiling Service 業務データ可視化向け
配車、配送、業務制約を入れる SDK単体ではなく独自ロジック併用 業務要件は別設計が必要

迷ったときの実践的な選び方

  • まず運転ナビとして成立させたいなら Navigation SDK から入る
  • 地図演出や重ね合わせが肝なら Maps SDK の設計を先に詰める
  • 目的地検索が重要なら Search SDK か Search API を最初から分離して考える
  • 業務車両向けなら SDK 選定だけでなく、業務制約をどこで処理するかを先に決める

要点のまとめ

Mapboxでカーナビを作る場合、中心はNavigation SDKです。 その上で、地図表現を作るMaps SDK、目的地入力を支えるSearch系機能、業務要件を支える独自ロジックを どう分担するかが設計の要点になります。 単純な乗用車ナビなら Navigation SDK を起点に進めるのが最短で、配送や特殊車両まで踏み込むなら Maps SDK とバックエンド設計を早めに並行検討したほうが崩れにくいです。
